# 需求文档

## 1. 项目边界

### 1.1 支持的语言
- 仅支持 Python 代码审查（v1.0 范围）
- 后续可扩展 JavaScript、Go 等

### 1.2 审查维度
系统从以下三个维度对代码进行审查：
- **安全 (Security)**：SQL 注入、命令注入、路径遍历、敏感信息硬编码、不安全随机数、反序列化漏洞等
- **性能 (Performance)**：时间复杂度、N+1 查询、不必要循环、内存泄漏风险、重复计算、低效数据结构等
- **风格 (Style)**：PEP8 命名规范、函数长度、注释质量、代码重复、魔法数字、函数职责单一性等

### 1.3 不做的事情
- 不审查非 Python 代码
- 不自动合并修复代码到用户仓库
- 不提供实时 IDE 插件（v1.0 只做 Web 界面）

---

## 2. 用户使用流程

```
用户打开 Streamlit 页面
  → 在输入框中粘贴代码
  → 点击"提交审查"按钮
  → 实时看到三个 Agent 并行审查进度（流式输出）
  → 看到汇总的审查结果（问题列表 + 评分）
  → 看到自动修复后的代码（diff 对比视图）
  → 等待沙箱验证自动执行修复后代码
  → 验证通过后，弹出确认框：
      - 用户点击"确认" → 输出最终报告
      - 用户输入修改意见 → 系统重新修复
  → 可查看历史审查记录
```

---

## 3. 输入输出规格

### 3.1 输入
| 字段 | 类型 | 说明 |
|------|------|------|
| 源代码 | string | 用户提交的 Python 代码，最大 10000 字符 |

### 3.2 输出（最终报告）
| 字段 | 类型 | 说明 |
|------|------|------|
| 原始代码 | string | 用户提交的原始代码 |
| 修复后代码 | string | 自动修复后的代码 |
| 问题列表 | list | 发现的所有问题（编号、维度、严重程度、位置、描述、建议） |
| 修复说明 | list | 每处修改对应的解释 |
| 修复前评分 | int | 0-100 综合评分 |
| 修复后评分 | int | 修复后预估综合评分 |
| 沙箱验证 | bool | 修复代码是否通过执行验证 |
| 重试次数 | int | 修复经历的重试次数 |
| 自然语言总结 | string | 整体评价和关键建议 |

---

## 4. 非功能性需求

- **安全性**：用户代码必须在 Docker 沙箱中隔离执行（网络隔离、内存 128MB、CPU 0.5 核、超时 10 秒）
- **响应速度**：三路并行审查确保总耗时接近单路审查时间
- **容错性**：失败自动重试最多 3 次，3 次后输出失败报告而非崩溃
- **可观测性**：每个节点执行有日志记录，前端可实时看到执行进度

---

## 5. 成功标准

- [ ] 能正确审查 Python 代码的三个维度
- [ ] 三路审查真正并行执行（日志可验证）
- [ ] 修复代码通过沙箱验证
- [ ] 失败时自动反思重试（上限 3 次）
- [ ] 验证通过后等待用户确认再输出
- [ ] Docker 沙箱实现网络/内存/CPU 隔离
- [ ] Streamlit 界面有流式进度显示
- [ ] 历史记录可查看
