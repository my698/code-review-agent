## 1. Pydantic `Field` 字段配置函数

### 是什么

`Field` 是 Pydantic（`from pydantic import BaseModel, Field`）提供的字段配置函数，写在 `BaseModel` 类属性里，用来给字段加**额外规则**（校验、默认值等）。

### 为什么需要它

- **自动数据校验**：不用手写 `if 0 <= x <= 100`，Pydantic 创建对象时自动检查
- **避免可变默认值陷阱**：Python 的 `list`、`dict` 作为默认值会导致所有实例共享同一个对象，`Field` 能解决

---

### 用法 1：`Field(default_factory=list)` — 安全空列表

```python
issues: list[Issue] = Field(default_factory=list)
```

- `default_factory` = 每次创建对象时调用的函数，生成新默认值
- `list` = Python 内置的 `list()`，每次调用创建新空列表
- **为什么不用 `= []`**：Python 的坑——直接写 `= []` 所有实例共享同一个列表对象。A 往里加东西，B 也会出现

---

### 用法 2：`Field(ge=0, le=100)` / `Field(ge=0)` — 数值范围校验

```python
score_before: int = Field(ge=0, le=100)   # 必须 0-100 之间
total_issues: int = Field(ge=0)            # 必须 ≥ 0
```

- `ge` = **g**reater than or **e**qual（大于等于，≥）
- `le` = **l**ess than or **e**qual（小于等于，≤）
- 如果 LLM 返回 `score_before: 150`，Pydantic 直接报 `ValidationError`，脏数据不会流入后续节点

---

### 用法 3：`Field(default_factory=lambda: {...})` — 安全空字典

```python
by_severity: dict[str, int] = Field(
    default_factory=lambda: {"critical": 0, "high": 0, "medium": 0, "low": 0}
)
```

- `lambda` = 匿名函数（一次性短函数，只写一行）
- `lambda: {"critical": 0, ...}` = 每次调用返回一个新字典
- **为什么不用 `= {"critical": 0}`**：同用法1，字典也是可变对象，直接写会所有实例共享一个字典

---

### 用法总结

| 写法 | 适用类型 | 作用 |
|------|---------|------|
| `Field(default_factory=list)` | `list` | 每个实例有独立空列表 |
| `Field(default_factory=lambda: {...})` | `dict` | 每个实例有独立空字典 |
| `Field(ge=0, le=100)` | `int` | 数值范围自动校验 |
| 直接 `= ""` | `str` | 字符串不可变，不存在共享陷阱，直接用 |

## 2. `Optional` 什么时候需要、什么时候不需要

### 核心判断：类型本身能不能表示"空"

| 类型 | 能表示"空"吗 | 需要 Optional 吗 | 例如 |
|------|:--:|:--:|------|
| `int` | ✅ `0` 本身就是合法值 | 不需要 | `retry_count: int = 0`，初始重试次数就是 0，不是"不知道" |
| `str` | ✅ `""` 本身就是合法值 | 不需要 | `reflection_notes: str = ""`，初始没有反思笔记就是空字符串 |
| `bool` | ✅ `False` 本身就是合法值 | 不需要 | `sandbox_passed: bool = False`，初始就是"尚未通过" |
| `list[T]` | ✅ `[]` 本身就是合法值 | 不需要 | `review_results = []`，初始就是"还没有审查结果" |
| `Pydantic 模型` | ❌ 没法用空对象替代，`CriticSummary()` 把所有字段填错会影响后续逻辑 | **需要** | `critic_summary: Optional[CriticSummary] = None` |
| `你的自定义对象` | ❌ 同 Pydantic，没有一个有意义的"空替身" | **需要** | `final_report: Optional[FinalReport] = None` |

### 为什么 `int` / `str` 不需要 Optional

```python
retry_count: int = 0                # 0 是合法的 int，语义就是"还没重试过"
reflection_notes: str = ""          # "" 是合法的 str，语义就是"没有反思笔记"
```

这些类型**自带**能表达"空"的值，直接拿来当初始状态就行。

### 为什么 Pydantic 模型需要 Optional

```python
critic_summary: Optional[CriticSummary] = None   # 初始时根本不存在，只能是 None
```

`CriticSummary` 是一个复杂的数据结构，里面有 `score_before`、`action_plan`、`by_severity` 等字段。如果硬塞一个 `CriticSummary()` 空对象，里面的 `score_before=0` 会被下游误解为"评分 0 分"（实际是还没评分）。**创建一个空壳会让虚假数据流入后续逻辑**，Bug 比不用 Optional 更严重。

### 一句话

**类型自带的零值（0 / "" / False / []）就是它的合法状态，不需要 Optional。复杂对象没有"有意义的空值"，只有"存在"和"不存在"两种状态，必须用 Optional + None。**

---

### `Optional` 语法补充

```python
cwe_id: Optional[str] = None
```

- `Optional[str]` = 这个字段可以是 `str` 类型，也可以是 `None`
- `= None` = 默认值为 `None`，LLM 返回的 JSON 里没这个字段也不报错
- 等于 `str | None = None`（Python 3.10+ 写法），但 TypedDict 用 `Optional` 兼容性更好

## 3. `str, Enum` — 字符串枚举

```python
class Severity(str, Enum):
    CRITICAL = "critical"
    HIGH = "high"
    MEDIUM = "medium"
    LOW = "low"
```

- 继承 `str` + `Enum`——序列化时直接是字符串 `"critical"`，不用 `.value`
- 继承纯 `Enum` 的话，存 JSON 会变成 `"Severity.CRITICAL"`，不好处理

## 4. Git 基本操作 — add / commit / push 的区别

三个操作都在做不同的事，都只发生在**本机**（除了 `push`）：

| 命令 | 发生在哪 | 干什么 |
|------|---------|------|
| `git add` | 本机 | 把文件改动加入**暂存区**，标记为"下次提交要包含" |
| `git commit` | 本机 | 把暂存区内容拍成一张**版本快照**，存入本地仓库 |
| `git push` | 推送到远程 | 把本地的 commit 推送到**GitHub**，别人才能看到 |

```
工作目录               暂存区                 本地仓库              GitHub
(改文件)  ──add──→  (待提交清单)  ──commit──→  (版本历史)  ──push──→  (远程仓库)
```

- `git add` 不是保存到本地硬盘（文件改动本来就在硬盘上），而是**选中**哪些改动要进下一次 commit
- `git commit` 才是真正"保存版本"，但只保存在本机
- `git push` 是把本机版本同步到 GitHub，不执行 push 的话 GitHub 上看不到任何变化

## 5. `Annotated[list[T], operator.add]` — LangGraph 并行写入防覆盖

### 场景问题

三个审查 Agent（security / performance / style）并行执行，跑完后各自返回：

```python
# security_reviewer 返回
{"review_results": [security_result]}

# performance_reviewer 返回
{"review_results": [perf_result]}

# style_reviewer 返回
{"review_results": [style_result]}
```

三个节点同时往 `review_results` 字段写数据。如果是普通字段，**最后完成的节点会覆盖前面所有结果**——比如 style 最后跑完，state 里只剩它的结果，安全审查和性能审查的成果全丢了。

### 解决方案

声明字段时用 `Annotated` 包装类型，`operator.add` 定义写入规则：

```python
from typing import Annotated
import operator

review_results: Annotated[list[ReviewResult], operator.add]
```

### 拆分理解

| 部分 | 是什么 | 干什么 |
|------|--------|--------|
| `Annotated` | `typing` 标准库的标记工具 | 告诉 LangGraph："这个字段的写入行为不 是用新值替换旧值" |
| `list[ReviewResult]` | 实际数据类型 | 字段存的是 `ReviewResult` 对象的列表 |
| `operator.add` | Python 内置的 `+` 运算符函数 | 定义自定义行为：`旧列表 + 新列表` |

- `Annotated[原始类型, 元数据1, 元数据2, ...]`——在原始类型上贴标签，不改变运行时类型
- `operator.add` 就是 `list.__add__`，等于 `[1] + [2]` → `[1, 2]`
- LangGraph 拿到节点返回的 `{"review_results": [新结果]}` 时，不替换原列表，而是执行 `旧列表 + [新结果]`

### 效果

```
初始: []
第1个写入: [] + [sec_result]  → [sec_result]
第2个写入: [sec_result] + [perf_result] → [sec_result, perf_result]
第3个写入: [sec_result, perf_result] + [style_result] → [sec_result, perf_result, style_result]
```

三个结果都在，谁也不丢。

### 三个审查节点返回格式必须统一

```python
# 每个节点返回单元素列表，才能正确触发追加
return {"review_results": [review_result]}  # ✅ 正确

# 不能直接返回 ReviewResult 对象
return {"review_results": review_result}    # ❌ operator.add 是 list+list，不是 list+obj
```

### `Annotated` 是 Python 标准库，不是 LangGraph 专属

```python
# 在纯 Python 代码中也可以用，本质是存元数据
from typing import Annotated, get_type_hints

x: Annotated[int, "单位: 厘米"] = 10
# 运行时 x 就是普通 int，元数据通过 get_type_hints(include_extras=True) 读取
```

LangGraph 在启动时读取 state 的类型标注，发现有 `Annotated[..., operator.add]` 的就自动切换为追加模式。

## 6. `sum(生成器表达式)` — 对序列逐元素计算后累加

### 语法形式

```python
sum(len(r.issues) for r in state['review_results'])
```

### 拆开理解

| 部分 | 是什么 | 干什么 |
|------|--------|--------|
| `sum(...)` | Python 内置函数 | 把括号里所有数值加起来 |
| `len(r.issues)` | 每次循环求的值 | 取当前 ReviewResult 的 issues 列表长度 |
| `for r in state['review_results']` | 生成器表达式 | 遍历每个元素，每次产出一个数值给 sum |

### 执行过程

```python
# 假设
review_results = [
    ReviewResult(issues=[...3个...]),   # r.issues 长度 = 3
    ReviewResult(issues=[...1个...]),   # r.issues 长度 = 1
    ReviewResult(issues=[...2个...]),   # r.issues 长度 = 2
]

# sum 内部等价于:
total = 0
for r in review_results:
    total += len(r.issues)   # total: 0→3→4→6
# 结果: 6
```

### 为什么不用方括号写法

```python
# 写法 A（生成器，不建中间列表）
sum(len(r.issues) for r in review_results)

# 写法 B（先建列表再求和）
sum([len(r.issues) for r in review_results])
```

两种都对，但 A 不建中间列表，省内存。数据量很小时没区别，数据量大时 A 更优。

### 注意：`sum` 有两个默认参数

```python
sum(iterable, start=0)  # start 默认是 0，一般不用改
```

所以 `sum(...)` 就是 `sum(..., 0)`，从 0 开始累加。

## 7. `"\n".join(列表)` — 用分隔符拼接字符串列表

### 语法形式

```python
"\n".join(issues_text)
```

### 记忆方式

**分隔符.join(列表)** — 分隔符放在前面，列表放在后面。分隔符只出现在元素**之间**，开头和结尾不插。

### 例子

```python
issues_text = [
    "行12 critical | SQL注入",
    "  建议: 改用参数化查询",
    "行5 medium | 命名不规范",
]

result = "\n".join(issues_text)
print(result)
```

输出：

```
行12 critical | SQL注入
  建议: 改用参数化查询
行5 medium | 命名不规范
```

- 元素 1 和元素 2 之间有 `\n`
- 元素 2 和元素 3 之间有 `\n`
- 开头和结尾没有多余的 `\n`

### 为什么是 `"\n".join()` 而不是 `issues_text.join("\n")`

这是 Python 的设计选择——`join` 是**字符串**的方法，不是列表的方法。原因是 `join` 只对字符串列表有意义，放在 `str` 类型上更合理：任何分隔符都能调用它。

### 常见分隔符

```python
", ".join(items)     # 逗号+空格: "a, b, c"
"".join(items)       # 无分隔符: "abc"
" | ".join(items)    # 竖线分隔: "a | b | c"
"\n\n".join(items)   # 空行分隔: 元素间留一行空白
```

### 注意：`join` 的参数必须是字符串列表

```python
"\n".join(["a", "b"])           # ✅ 正确
"\n".join([1, 2, 3])            # ❌ TypeError: int 转不了 str
"\n".join(str(x) for x in ...)  # ❌ join 不接受生成器，只接受列表/元组
```

如果元素不是字符串，要先用列表推导式转成字符串列表再 `join`。

## 8. `Send` API — LangGraph 并行分发与 state 机制

### 为什么需要 Send

三个审查员（security / performance / style）之间没有数据依赖，不需要串行等待。LangGraph 提供了两种并行方式：

| 方式 | 何时决定分支 | 每分支不同数据 | 适用场景 |
|------|:--------:|:--------:|------|
| 多条 `add_edge` | 写代码时写死 | ❌ 不能 | 分支固定、每分支数据相同 |
| `list[Send]` | 运行时动态 | ✅ 可以 | 分支动态、每分支数据可定制 |

`Send` 解决的核心问题是**动态**：分支数量和每分支数据可以在运行时根据 state 内容决定，而不是写死在代码里。

### 语法

```python
from langgraph.types import Send

def fanout_to_reviewers(state: AgentState) -> list[Send]:
    return [
        Send("security_reviewer", {}),
        Send("performance_reviewer", {}),
        Send("style_reviewer", {}),
    ]

workflow.add_conditional_edges(
    "code_parser",
    fanout_to_reviewers,
    ["security_reviewer", "performance_reviewer", "style_reviewer"],
)
```

### `Send` 是什么

`Send` 是一个**类**，从 `langgraph.types` 导入。`Send("节点名", {...})` 创建一个 `Send` 实例对象。既然是对象，就可以放进 `list[Send]` 里。

```python
Send("目标节点名", {"key": value})
#    参数1: str    参数2: dict（可选）
```

### `Send` 第二个参数的作用机制（重点）

**核心规则：LangGraph 中只有节点 `return` 的字典会真正写入主 state。`Send` 的第二个参数只在当前分支的 state 副本上生效，不写回主 state。**

```
主 state
   │
   ├── 分支1: state副本 = 主state + {"focus": "注入漏洞"}   ← security_reviewer 看到的
   │   return {"review_results": [...]}   ← 只返回 review_results
   │   分支副本销毁，focus 丢弃
   │
   ├── 分支2: state副本 = 主state + {"focus": "N+1查询"}   ← performance_reviewer 看到的
   │   return {"review_results": [...]}
   │   分支副本销毁，focus 丢弃
   │
   └── 分支3: state副本 = 主state + {"focus": "PEP 8"}     ← style_reviewer 看到的
       return {"review_results": [...]}
       分支副本销毁，focus 丢弃

合并回主 state：只合并 return 里的东西
    review_results = [...] + [...] + [...]  ← operator.add 追加 ✅
    focus = 没被任何分支 return，保持原样（None）✅
```

### 关键结论

| 操作 | 影响范围 | 是否写回主 state |
|------|---------|:---------:|
| `Send("xx", {"k": v})` | 当前分支的 state 副本 | ❌ 不写回 |
| `return {"k": v}` | 当前分支 state → 主 state | ✅ 写回 |

**所以审查员内部看到的 state 可能不是"纯粹的主 state"**，而是主 state + Send 临时覆盖后的分支专属副本。但只有 `return` 的内容会被合并回主 state，Send 的临时覆盖用完就扔掉。

### 字段必须在 `AgentState` 中声明

`Send` 不能凭空创造 state 字段。如果第二个参数里塞了一个 `AgentState` 不存在的 key，LangGraph 编译时可能报警告或被忽略。

```python
# state.py 中必须先声明
class AgentState(TypedDict):
    focus: Optional[str]  # 声明字段，默认 None

# 然后 Send 才能用
Send("security_reviewer", {"focus": "注入漏洞"})
```

- 声明了但不用赋初值，默认就是 `None`
- `Send` 只是临时分行值给对应节点看，用完即弃
- 节点 `return` 时才真正写回主 state

### 什么场景需要传不同的第二个参数

当不同分支需要不同的"内部上下文"时：

```python
def fanout(state) -> list[Send]:
    if state["code_analysis"].language == "python":
        return [
            Send("security_reviewer", {"focus": "注入漏洞和权限校验"}),
            Send("performance_reviewer", {"focus": "时间复杂度与IO"}),
            Send("style_reviewer", {"focus": "PEP 8 和类型注解"}),
        ]
    else:
        return [
            Send("security_reviewer", {}),  # 其他语言只做安全审查
        ]
```

当前项目中三个 reviewer 都用同样的 `code_analysis` + `original_code`，所以可以简写 `Send("xxx", {})`。

### `Send` 的参数名记忆

```
Send("目标节点", {"state字段": "给该字段的分支临时值"})
```

- 第一个参数：告诉 LangGraph "下个节点是谁"
- 第二个参数：给分支看的临时便签，用完即弃

## 9. `Send` 第二个参数——正确理解（纠正第 8 节错误）

> ⚠️ 第 8 节关于 Send 第二个参数的描述有误，本节纠正。保留第 8 节原样，方便对比学习。

### 错误理解（第 8 节写的）

> 错误认为：Send 第二个参数合并到主 state 副本上，分支能读到主 state 的所有字段 + Send 覆盖的字段。即分支 state = 主 state + Send 覆盖。

### 实际运行时验证

`fanout_to_reviewers` 只传了 `code_analysis`：

```python
Send("security_reviewer", {"code_analysis": state["code_analysis"]})
```

目标节点 `security_reviewer` 尝试读 `state["original_code"]`，报错：

```
KeyError: 'original_code'
During task with name 'security_reviewer'
```

说明分支 state 中**没有** `original_code`。如果分支继承了主 state，不应该报这个错。

### 正确理解

**Send 的第二个参数就是目标分支的全部 state 输入，不会自动继承主 state。**

```
主 state（有 13 个字段：original_code, code_analysis, review_results, ...）
    │
    ├── Send("security_reviewer", {"code_analysis": ..., "original_code": ...})
    │   └── 分支 state = {"code_analysis": ..., "original_code": ...}
    │       只有 Send 传的两个字段，主 state 其他 11 个字段不会自动带过来
    │
    ├── Send("performance_reviewer", {"code_analysis": ..., "original_code": ...})
    │   └── 分支 state = {"code_analysis": ..., "original_code": ...}
    │
    └── Send("style_reviewer", {"code_analysis": ..., "original_code": ...})
        └── 分支 state = {"code_analysis": ..., "original_code": ...}
```

### 正确结论

| 之前的理解（错误） | 实际行为（正确） |
|---|---|
| 分支 state = 主 state + Send 覆盖 | 分支 state = Send 第二个参数（仅此而已） |
| Send 只传新增/覆盖字段就行，其余继承主 state | Send 必须显式传目标节点需要的**每一个**字段 |
| `Send("xx", {"focus": "注入"})` 分支还能读到 `original_code` | 分支只能读到 `focus`，读不到 `original_code` |
| `Send("xx", {})` 分支拿到完整主 state | `Send("xx", {})` 分支拿到**空 state**，啥也读不到 |

### 为什么之前会误解

之前把 LangGraph Send 的行为跟普通函数调用混淆了。直觉上"图里所有节点共享同一个 state"，以为 Send 只是在主 state 上叠加。实际上 Send 创建的是独立分支，每个分支有自己的 state 副本，副本的初始内容完全由 Send 第二个参数决定。

### `return` 的行为不变

节点 `return` 的字典仍然会合并回主 state，这个理解没有错。只是 Send 的"输入"环节理解错了。

```python
def security_reviewer(state: AgentState) -> dict:
    # state 里只有 Send 传来的字段
    result = ...  # 用 state["code_analysis"] 和 state["original_code"] 做审查
    return {"review_results": [result]}  # 这个会合并回主 state ✅
```

### 修正后的代码

```python
def fanout_to_reviewers(state: AgentState) -> list[Send]:
    return [
        Send("security_reviewer", {
            "code_analysis": state["code_analysis"],
```
